# خرید کمتر از مقدار trade_amount

## Context (چرا این تغییر)

انتظار کاربر: `trade_amount` فقط **سقف** باشد (مثلاً ۵۰). اگر موجودی والت کمتر از این بود، باید با همان مقدارِ کمتر معامله انجام شود — به شرطی که فرصت هنوز سودده باشد. (کاربر خودش بیرونی هر ~۵ ثانیه `/run` را تریگر می‌کند، پس سیکلِ تکرار بیرون از پایتون است: هر تریگر یک سفارش تا سقفِ ۵۰، و با تکرار، موجودی پله‌پله خالی می‌شود.)

رفتار فعلی: وقتی والت کمتر از `trade_amount` دارد، **هیچ معامله‌ای انجام نمی‌شود** (تأییدشده توسط کاربر).

علت ریشه‌ای — یک خط در [`apply_balance_limit`](../arbitrage_main/engine.py#L375):

```python
# engine.py:417
if opp["net_pct"] < self.min_pct or opp["net_usdt"] < self.min_usdt:
    ...
    return None
```

این گیت **بعد از** کپ‌شدنِ اندازه با موجودی والت اجرا می‌شود. چون سود مطلق خطی با اندازه است:

```python
spread_profit = amount * (eff_sell - eff_buy)   # خطی با amount  →  net_usdt خطی با amount
```

ولی `net_pct` نسبت به اندازه **ثابت** است (هم اسپرد و هم کارمزدهای انتقالِ flat به‌صورت درصدی روی `amount` پخش می‌شوند و `amount` حذف می‌شود — به [`_recompute_profit`](../arbitrage_main/engine.py#L301) نگاه کن).

نتیجه: کپ‌شدن از ۵۰ به (مثلاً) ۲۰ ⇒ `net_usdt` به ~۰٫۴ برابر آب می‌رود ⇒ اگر زیر `min_profit_usdt` بیفتد، کل فرصت `return None` می‌شود، با اینکه `net_pct` تغییری نکرده. عملاً `min_profit_usdt` به‌طور ضمنی برای اندازه‌ی کاملِ ۵۰ کالیبره شده و هر کپِ ناشی از کمبودِ موجودی، معامله را می‌کُشد — یعنی سفارش‌های آخرِ کوچک‌تر-از-۵۰ هیچ‌وقت اجرا نمی‌شوند.

**تصمیم کاربر:** `min_profit_usdt` نباید درصدی/مقیاسی رفتار کند؛ همان مقداری که وارد شده باید ملاک باشد، فارغ از اینکه معامله نهایتاً ۵۰ دلار شود یا ۲۵ دلار. یعنی آستانه‌ی مطلق فقط **یک‌بار** سرِ ارزیابیِ فرصت روی اندازه‌ی پیکربندی‌شده چک شود، نه دوباره روی اندازه‌ی کپ‌شده.

## نتیجه‌ی مطلوب

- `trade_amount` = سقف. اگر والت کمتر داشت، با همان کمتر معامله شود.
- گیت مطلق `min_profit_usdt` همان‌جا که الان هست — سرِ `qualifies` در [`scan`](../arbitrage_main/engine.py#L478) روی اندازه‌ی پیکربندی‌شده — اعمال می‌شود و بس.
- بعد از کپِ والت فقط گیتِ درصدیِ مقیاس‌ناوردای `min_pct` دوباره چک شود (که عملاً بدون تغییر است و فقط مقابلِ حرکتِ قیمت بین scan و execute محافظت می‌کند).

## تغییر (فقط [`arbitrage_main/engine.py`](../arbitrage_main/engine.py))

در `apply_balance_limit`، گیتِ بعد از کپ (خطوط ~۴۱۷–۴۲۳) به این صورت اصلاح شود تا فقط گیتِ درصدی باقی بماند:

```python
opp["amount"]          = capped
opp["balance_limited"] = capped < prev
self._recompute_profit(opp, cfg)

# فقط گیتِ درصدیِ مقیاس‌ناورد دوباره چک می‌شود. آستانه‌ی مطلقِ min_profit_usdt
# قبلاً سرِ qualify در scan() روی اندازه‌ی پیکربندی‌شده‌ی trade_amount چک شده؛
# اعمالِ دوباره‌اش روی اندازه‌ی کپ‌شده‌ی والت، هر سفارشِ کوچک‌تر از سقف را بی‌صدا
# می‌کشد، با اینکه لبه‌ی درصدی همان است. min_profit_usdt همان‌طور که وارد شده
# محترم است، نه بازمقیاس‌شده با اندازه.
if opp["net_pct"] < self.min_pct:
    self.log.warning(
        "[%s] %s->%s SKIP after balance cap: amount=%.6g  net=%.3f%%  "
        "below gate (min %.3f%%)",
        opp["pair"], buy_ex, sell_ex, capped, opp["net_pct"], self.min_pct)
    return None
```

تنها تفاوت با نسخه‌ی فعلی: حذفِ شرطِ `or opp["net_usdt"] < self.min_usdt` (و پاک‌سازیِ پیام لاگ از فیلدهای مطلق). گیتِ `capped <= 0` در خط ۴۰۵ دست‌نخورده می‌ماند تا فرصت‌های بدونِ موجودیِ قابل‌فاند هنوز رد شوند.

### چرا این درست است و دو-بار-شماری ندارد
- در [`scan`](../arbitrage_main/engine.py#L478)، `qualifies = net_pct >= min_pct and net_usdt >= min_usdt` روی اندازه‌ی اولیه (`min(trade_amount, ask_vol, bid_vol)`) محاسبه می‌شود — این همان «روی مقدار وارد‌شده» است که کاربر می‌خواهد.
- چون `net_pct` با اندازه ثابت است، چکِ دوباره‌ی `min_pct` بعد از کپ تقریباً همیشه همان نتیجه را می‌دهد؛ فقط برای امنیت در برابر تغییرِ قیمت بین scan و execute نگه داشته می‌شود.

### تبصره درباره‌ی خالی‌شدنِ کیف
چون `capped = min(prev, 0.95*buy_cap, 0.95*sell_cap)`، اندازه وقتی شروع به کوچک‌شدن می‌کند که طرفِ محدودکننده به زیرِ ~۵۲٫۶ برسد (چون `0.95*cap ≥ 50`). از آن‌جا اندازه پله‌پله کم می‌شود (۵۰ → ۳۸ → ۲۰ → ...) و با این فیکس همه‌ی این سفارش‌های کوچک اجرا می‌شوند. ضریبِ `balance_safety_factor=0.95` همیشه ~۵٪ ته‌مانده‌ی دست‌نخورده باقی می‌گذارد (بافرِ ضدِ ردِّ سفارش)؛ پس کیف تا صفرِ مطلق خالی نمی‌شود. اگر خالی‌شدنِ کامل لازم شد، این یک تنظیمِ کانفیگ جداست (`balance_safety_factor` نزدیک‌تر به ۱٫۰)، نه بخشی از این فیکس.

## یادداشتِ ثانویه (خارج از اسکوپِ این فیکس، فقط ثبت برای آینده)

در [`scan_once`](../arbitrage_main/engine.py#L636) فقط `best_candidate` (بالاترین `net_pct`) امتحان می‌شود. اگر همان یک کاندید بعد از کپِ والت رد شود (مثلاً موجودیِ سمتِ موردنیازش صفر باشد)، روی بقیه‌ی فرصت‌های قابل‌فاندِ همان pair **fallback ندارد**. این عاملِ سمپتومِ فعلی نیست (تا وقتی والت روی legهای درست مقداری داشته باشد، `capped > 0` و معامله انجام می‌شود)، ولی یک شکافِ مقاومت است. در صورت تمایلِ کاربر می‌توان بعداً حلقه‌ی `scan_once` را طوری تغییر داد که در صورت `None`-شدنِ best، فرصتِ قابل‌فاندِ بعدی را امتحان کند.
